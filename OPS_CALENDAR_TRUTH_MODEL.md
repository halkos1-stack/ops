# OPS CALENDAR TRUTH MODEL

## Σκοπός

Το αρχείο αυτό κλειδώνει τη νέα λογική του OPS σε μία καθαρή μορφή:

**το σύστημα δεν υπολογίζει readiness verdict**

Αντί γι' αυτό:

- προβάλλει την αλήθεια του ακινήτου
- οργανώνει αυτή την αλήθεια γύρω από το ημερολόγιο ακινήτου
- επιτρέπει στον διαχειριστή να πάρει την επιχειρησιακή απόφαση

---

## 1. Canonical αρχές

### 1. Η κύρια προβολή της αλήθειας είναι το ημερολόγιο ακινήτου

Το ημερολόγιο ακινήτου πρέπει να είναι το σημείο όπου βλέπουμε:

- κρατήσεις
- εργασίες
- αναλώσιμα
- ελλείψεις
- ζημιές
- βλάβες
- θέματα / conditions
- χρονικά σημεία που χρειάζονται ενέργεια

### 2. Η αλήθεια παραμένει αναλυωτη

Το σύστημα δεν πρέπει να εξαφανίζει ή να υπεραπλουστεύει την πραγματική κατάσταση του ακινήτου σε μία τελική ένδειξη που αντικαθιστά την κρίση του διαχειριστή.

### 3. Ο διαχειριστής αποφασίζει

Ο διαχειριστής είναι αυτός που αποφασίζει:

- αν απαιτείται εργασία
- πότε απαιτείται εργασία
- τι τύπου εργασία απαιτείται
- ποιο θέμα παρακολουθείται
- ποιο θέμα κλιμακώνεται
- ποιο θέμα κλείνει

### 4. Η εργασία είναι execution layer

Η εργασία δεν είναι η κύρια αλήθεια του ακινήτου.
Είναι το μέσο εκτέλεσης που δημιουργεί ο διαχειριστής όταν το κρίνει απαραίτητο.

### 5. Οι κρατήσεις είναι input layer

Οι κρατήσεις μπαίνουν στο OPS ως είσοδος δεδομένων.
Δεν αποτελούν από μόνες τους το τελικό operational meaning του ακινήτου.

---

## 2. Τι πρέπει να δείχνει το property calendar model

Για κάθε σχετική ημέρα ή χρονικό σημείο, το μοντέλο πρέπει να προβάλλει:

- occupancy / movement κράτησης
- open ή scheduled tasks
- active alerts
- shortages αναλωσίμων
- open issues / damages / conditions
- χρήσιμες ομαδοποιήσεις προβολής για τον διαχειριστή

Το αποτέλεσμα αυτού του μοντέλου πρέπει να είναι **descriptive projection** και όχι readiness decision.

---

## 3. Τι δεν πρέπει να κάνει το σύστημα

Το σύστημα δεν πρέπει να:

- αποφασίζει μόνο του ότι ένα ακίνητο είναι έτοιμο ή μη έτοιμο
- αντικαθιστά τη διαχειριστική κρίση με readiness score ή readiness state ως κύρια αλήθεια
- θεωρεί ότι η απουσία εργασίας σημαίνει αυτομάτως καθαρή κατάσταση
- θεωρεί ότι η ύπαρξη κράτησης δημιουργεί από μόνη της την τελική επιχειρησιακή σημασία

---

## 4. Αρχές δημιουργίας εργασιών

Κλειδωμένη αρχή:

**Ο διαχειριστής μπορεί να δημιουργήσει οποιαδήποτε εργασία, οποτεδήποτε, για οτιδήποτε προκύπτει.**

Ενδεικτικά triggers:

- άφιξη / αναχώρηση
- βλάβη
- ζημιά
- έλλειψη αναλωσίμων
- προληπτικός έλεγχος
- επιθεώρηση
- ad hoc λειτουργική ανάγκη

---

## 5. Σωστή γλώσσα στον κώδικα και στα docs

### Επιθυμητή γλώσσα

- property calendar
- operational truth
- day snapshot
- booking movement
- task coverage
- issues / damages / shortages
- manager decision
- execution layer

### Γλώσσα που πρέπει να αποσυρθεί από τον ενεργό πυρήνα

- readiness-first
- readiness hub
- not_ready / borderline / ready ως κύρια επιχειρησιακή αλήθεια
- readiness verdict
- readiness snapshot ως τελικό αποτέλεσμα απόφασης

## 7. Κανόνας μετάβασης

Σε μεταβατική φάση μπορεί να παραμένουν legacy πεδία ή labels για λόγους συμβατότητας.

Όμως αυτά πρέπει να αντιμετωπίζονται ως:

- προσωρινά
- μη canonical
- προς αφαίρεση από τον ενεργό πυρήνα

Ο canonical πυρήνας του OPS πρέπει να ευθυγραμμιστεί πλήρως με τη λογική:

**calendar truth first, manager decision next**
